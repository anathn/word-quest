### Issue #1: set_voice() Method Missing Validation

- **Severity:** 🔴 Critical
- **Location:** `src/components/tts_engine.py:180` and `src/components/tts_manager.py:216`
- **Problem:** The `set_voice()` method accepts any voice_id string without validating it exists in the available voices list, potentially causing silent failures or future errors.
- **Impact:** If an invalid voice_id is set, TTS may fail silently or crash later when pyttsx3 tries to use the invalid voice. This breaks accessibility for users who depend on TTS.
- **Evidence:** Story requirements specify "Voice selection" in TTS settings, which implies validation against available voices.
- **Recommended Fix:** Validate voice_id against available voices before setting:

```python
# In PlatformTTSWrapper.set_voice():
def set_voice(self, voice_id: str) -> bool:
    """Set the voice to use"""
    if not self.initialized or not self.engine:
        return False
    
    try:
        # Validate voice exists
        voices = self.get_available_voices()
        if not any(voice['id'] == voice_id for voice in voices):
            logger.error(f"Voice '{voice_id}' not found in available voices")
            return False
        
        self.engine.setProperty('voice', voice_id)
        self._current_voice_id = voice_id
        logger.info(f"TTS voice set to: {voice_id}")
        return True
    except Exception as e:
        logger.error(f"Error setting voice: {e}")
        return False
```

### Issue #4: Potential Race Condition in TTS Queue Processing

- **Severity:** 🟡 Medium
- **Location:** `src/components/tts_manager.py:_speaker_loop()` and `speak_word()`
- **Problem:** The `speak_word()` method adds to the queue while holding a lock, but the `_speaker_loop` retrieves from queue without the lock during the blocking get() call. This could theoretically lead to race conditions or lost items in certain edge cases.
- **Impact:** In rare cases, speech items could be lost or duplicated, causing unexpected behavior for TTS-dependent users.
- **Recommended Fix:** Ensure all queue operations use consistent locking or use thread-safe queue operations:

```python
# In TTSManager._speaker_loop():
def _speaker_loop(self) -> None:
    """Background thread that processes speech queue"""
    logger.debug("TTS speaker loop started")
    
    while self._running:
        try:
            # Get text from queue with timeout (thread-safe)
            text = self.speech_queue.get(timeout=0.2)
            
            with self._lock:  # Add lock around state modifications
                self._current_text = text
                self._is_speaking = True
            
            # ... rest of speak processing ...
            
            # Mark as not speaking (with lock)
            with self._lock:
                self._is_speaking = False
                self._current_text = None
            
        except Empty:
            continue
        except Exception as e:
            logger.error(f"TTS speaker loop error: {e}")
            continue
```

### Issue #6: Missing Speed Setting Persistence in TTSManager

- **Severity:** 🟡 Medium
- **Location:** `src/components/tts_manager.py`
- **Problem:** When speed is changed via `set_speed()`, the current TTS session doesn't apply the new speed until the next utterance. This creates a delay in user feedback.
- **Impact:** Poor user experience - parents adjust speed slider but don't hear the change immediately.
- **Recommended Fix:** Apply speed changes immediately to running engine:

```python
def set_speed(self, speed: float) -> None:
    """Set speech speed with immediate application"""
    speed = max(0.5, min(2.0, speed))
    self.settings.speed = speed
    
    # Apply to engine immediately if initialized
    if self.engine and self.engine.initialized:
        try:
            base_rate = self.engine._base_rate
            adjusted_rate = int(base_rate * speed)
            self.engine.engine.setProperty('rate', adjusted_rate)
        except Exception as e:
            logger.error(f"Failed to apply speed immediately: {e}")
    
    self.config_manager.settings = self.settings
    self.config_manager.save()
    logger.debug(f"TTS speed set to {speed}x")
```

### Issue #7: Missing TTS Initialization Performance Measurement

- **Severity:** 🟡 Medium
- **Location:** `src/components/tts_manager.py`
- **Problem:** Story requires "Performance: TTS initialization <2 seconds" but there's no timing/metrics captured to verify this requirement is met.
- **Impact:** Cannot verify non-functional performance requirement.
- **Recommended Fix:** Add initialization timing:

```python
def __init__(self, config_manager: Optional[TTSConfigManager] = None):
    self.engine: TTSEngine = create_tts_engine()
    self.config_manager = config_manager or TTSConfigManager()
    self.config_manager.load()
    self.settings: TTSSettings = self.config_manager.settings
    
    # Measure initialization time
    import time
    init_start = time.time()
    self._initialized = self.engine.initialize()
    init_time = time.time() - init_start
    
    if init_time > 2.0:
        logger.warning(f"TTS initialization took {init_time:.2f}s (exceeds 2s target)")
    else:
        logger.info(f"TTS initialized in {init_time*1000:.0f}ms")
```
