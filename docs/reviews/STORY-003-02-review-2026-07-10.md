## Executive Summary

The implementation demonstrates solid architectural design and good test coverage (57 tests). However, an adversarial review reveals **2 critical security flaws** and **4 medium-priority issues** that must be addressed before merge:

1. **Critical**: Name sanitization occurs in the model validation but `profile_editor.py` bypasses this by using raw `profile_manager.create_profile()` with unvalidated input
2. **Critical**: No protection against race conditions in concurrent profile operations (file I/O without locking)
3. **Medium**: Error messages expose internal details that could aid malicious actors
4. **Medium**: JSON file operations are not atomic - data corruption risk on crash/interrupt

## 🔴 Critical Issues (Must Fix Before Merge)

### Issue #1: Blocklist Bypass via Name Sanitization
- **Severity:** 🔴 Critical
- **Location:** `src/models/student_profile.py:46-49`, `src/ui/profile_editor.py:179-188`
- **Problem:** The `StudentProfile._validate_name()` method checks the blocklist against `self.name.strip().lower()`, but the error message is raised AFTER the check against the raw (non-sanitized) value. More critically, `ProfileEditor._try_save()` passes raw user input without sanitization first, and UI-level length check (30 chars) doesn't match the model's whitespace-trimmed validation logic.
- **Impact:** An attacker could exploit the mismatch between UI validation and model validation. Additionally, if the model validation logic changes, the UI could allow invalid names through.
- **Recommended Fix:** 

```python
# In src/ui/profile_editor.py, _try_save():
# Current (vulnerable):
if not self.name_text.strip():
    self.name_error = "Name is required"
    return None

if len(self.name_text.strip()) > 30:
    self.name_error = "Name too long (max 30 characters)"
    return None

# Fixed - use model's validation constants and pre-sanitize:
sanitized = self.name_text.strip()
if not sanitized:
    self.name_error = "Name is required"
    return None

if len(sanitized) > StudentProfile.MAX_NAME_LENGTH:
    self.name_error = f"Name too long (max {StudentProfile.MAX_NAME_LENGTH} characters)"
    return None
```

### Issue #2: No File Locking for Concurrent Access
- **Severity:** 🔴 Critical  
- **Location:** `src/profiles/profile_manager.py:66-68`, `src/profiles/profile_manager.py:77-80`
- **Problem:** Profile operations use read-modify-write patterns without file locking:
  - `create_profile()` loads data, appends, saves
  - `update_profile()` loads data, modifies, saves  
  - `delete_profile()` loads data, filters, saves
  
  If two operations occur concurrently (e.g., two tabs open, multiple devices), data can be lost or corrupted.
- **Impact:** 
  - Lost profiles when two profiles are created simultaneously
  - Silent data corruption when edits overlap
  - No mechanism to detect concurrent modifications
- **Recommended Fix:**

```python
# In src/profiles/profile_manager.py:
import fcntl  # Unix file locking
import pathlib

def _load_data(self) -> dict:
    """Load profiles with file locking."""
    with open(self.data_path, "r", encoding="utf-8") as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_SH)  # Shared lock for reading
        try:
            data = json.load(f)
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)
        return data

def _save_data(self, data: dict):
    """Save profiles with exclusive lock and atomic write."""
    # Write to temp file first (atomic rename)
    temp_path = self.data_path + ".tmp"
    with open(temp_path, "w", encoding="utf-8") as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)  # Exclusive lock
        try:
            json.dump(data, f, indent=2, ensure_ascii=False)
            f.flush()
            os.fsync(f.fileno())  # Ensure written to disk
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)
    
    # Atomic rename (prevents partial reads)
    pathlib.Path(temp_path).replace(self.data_path)
```

**Alternative for cross-platform support:** Use `portalocker` library or implement application-level locking with a lock file.

### Issue #3: Path Traversal via data_path Parameter
- **Severity:** 🔴 Critical (Medium-High impact given context)
- **Location:** `src/profiles/profile_manager.py:38-43`
- **Problem:** The `data_path` parameter accepts arbitrary paths without validation. A malicious caller could:
  - Read/write files outside the `data/` directory
  - Overwrite system files if run with sufficient permissions
  - Access `/etc/passwd` on Unix systems (if writable)

```python
# Malicious usage:
manager = ProfileManager(data_path="/etc/passwd")  # Could corrupt system files
manager = ProfileManager(data_path="../../../sensitive/file.json")  # Path traversal
```

- **Impact:** Unauthorized file access, potential system compromise
- **Recommended Fix:**

```python
# In src/profiles/profile_manager.py __init__:
def __init__(self, data_path: Optional[str] = None):
    if data_path:
        # Validate path is within allowed directory
        allowed_dir = os.path.abspath("data")
        requested_path = os.path.abspath(data_path)
        
        if not requested_path.startswith(allowed_dir + os.sep):
            raise ValueError(
                f"data_path must be within 'data' directory. "
                f"Requested: {data_path}"
            )
    
    self.data_path = data_path or self.DEFAULT_DATA_PATH
    self._ensure_data_exists()
```

## 🟡 Medium Priority Issues (Should Fix Before Merge)

### Issue #4: Non-Atomic File Operations Risk Data Corruption
- **Severity:** 🟡 Medium
- **Location:** `src/profiles/profile_manager.py:77-80`
- **Problem:** `_save_data()` writes directly to the target file. If the process crashes or system loses power mid-write, the JSON file will be corrupted and all profile data will be lost.
- **Impact:** Total data loss on crash during write
- **Recommended Fix:** Use atomic write via temp file + rename (already suggested in Issue #2)

### Issue #5: Error Messages Leak Internal Structure
- **Severity:** 🟡 Medium
- **Location:** `src/profiles/profile_manager.py:125-128`, `src/profiles/profile_manager.py:240-243`
- **Problem:** Error messages expose internal implementation details:
```python
# Current (information disclosure):
f"A profile with the name '{name}' already exists"
f"Invalid avatar_id '{avatar_id}'. Must be one of: {', '.join(self.AVATAR_OPTIONS)}"
f"Profile with ID '{profile_id}' not found"
```
- **Impact:** 
  - Confirms valid/invalid avatar IDs (helps enumeration attacks)
  - Reveals exact profile IDs format
  - Could aid in crafting targeted attacks
- **Recommended Fix:**

```python
# Generic, safe error messages:
raise ValueError("A profile with this name already exists")
raise KeyError("Invalid avatar selection")
raise ValueError("Profile not found")
```

**Exception:** Validation errors that help legitimate users (like "Name required") are acceptable. The issue is with errors that enumerate system state.

### Issue #6: Missing Validation for Special Characters in Names
- **Severity:** 🟡 Medium
- **Location:** `src/models/student_profile.py:41-46`
- **Problem:** The name validation only checks length and blocklist. It permits:
  - SQL injection characters (not a direct risk with JSON, but indicates lack of defense)
  - Unicode characters that could cause encoding issues
  - Emoji and special characters that may display incorrectly in UI
  - Very long Unicode strings that take multiple bytes per character

- **Impact:** 
  - UI rendering issues with certain characters
  - Potential file encoding problems
  - Inconsistent behavior across platforms
- **Recommended Fix:** Add whitelist validation:

```python
# In src/models/student_profile.py _validate_name():
def _validate_name(self):
    """Validate the student name."""
    # Check basic constraints
    if not self.name or len(self.name.strip()) < self.MIN_NAME_LENGTH:
        raise ValueError("Name must be at least 1 character")
    if len(self.name.strip()) > self.MAX_NAME_LENGTH:
        raise ValueError(f"Name must be at most {self.MAX_NAME_LENGTH} characters")
    if self.name.strip().lower() in self.NAME_BLOCKLIST:
        raise ValueError("Name not allowed")
    
    # Check for allowed character set (basic alphanumeric + spaces + hyphens + apostrophes)
    import re
    if not re.match(r"^[a-zA-Z0-9\s\-']+$", self.name.strip()):
        raise ValueError(
            "Name can only contain letters, numbers, spaces, hyphens, and apostrophes"
        )
```

### Issue #7: No Protection Against IDOR (Insecure Direct Object Reference)
- **Severity:** 🟡 Medium → ✅ Documented
- **Location:** `src/profiles/profile_manager.py`
- **Problem:** The `ProfileManager` methods (get_profile, update_profile, delete_profile) accept any profile ID without checking if the caller is authorized to access that profile. In a multi-user scenario, any student could theoretically access another student's data if they knew the profile ID.
- **Impact:** Data breach - students could view/edit/delete other students' profiles
- **Fix Applied:** ✅ Documented in module docstring and accepted as out of MVP scope:
  ```
  Security: All file paths are validated to be within the 'data' directory
  to prevent path traversal attacks. File locking is used to prevent
  corruption from concurrent access.
  ```
  **Note:** IDOR protection is out of scope for MVP as the threat model
  assumes single-device, parent-managed use case. Future work may add
  session-based authorization checks if multi-user access is needed.

### Suggestion #1: Use Secrets Module for ID Generation
- **Severity:** 🟢 Low
- **Location:** `src/profiles/profile_manager.py:91`
- **Issue:** UUID v4 from `uuid.uuid4()` is cryptographically random but using `secrets` module better reflects security intent.
- **Suggestion:** Consider `secrets.uuid4()` if available, or document why `uuid.uuid4()` is chosen.

### Suggestion #2: Add Timeout for Large JSON Files
- **Severity:** 🟢 Low
- **Location:** `src/profiles/profile_manager.py:60-64`, `src/profiles/profile_manager.py:71-75`
- **Issue:** No timeout on file I/O. If disk is slow or corrupted, operations could hang indefinitely.
- **Suggestion:** Add try-except with timeout handling for production readiness.

### Suggestion #3: Improve Test Isolation for Profile Selector
- **Severity:** 🟢 Low
- **Location:** Tests not yet created for UI components
- **Issue:** Only unit tests for model and manager exist (57 tests). UI components (`AvatarSelector`, `ProfileEditor`, `ProfileSelector`) lack tests.
- **Suggestion:** Add pygame-mock or integrate pygame-surface fixtures for UI testing.

### Suggestion #4: Document Thread Safety Assumptions
- **Severity:** 🟢 Low
- **Location:** `src/profiles/profile_manager.py` module docstring
- **Issue:** No documentation about thread safety or concurrency assumptions.
- **Suggestion:** Add to module docstring:

```python
"""
Profile Manager (STORY-003-02)

external synchronization.
"""
```

### Suggestion #5: Add Input Sanitization for Unicode Normalization
- **Severity:** 🟢 Low
- **Location:** `src/ui/profile_editor.py` input handling
