## Overview

Implement password protection for parent settings to ensure student data privacy and prevent unauthorized access to word list management and progress viewing features.

## User Story

As a parent, I want the parent settings to be password-protected so that my child cannot accidentally or intentionally modify word lists or view sensitive information.

## Acceptance Criteria

### Functional Requirements

- [ ] Main menu includes "Parent Settings" button
- [ ] Clicking "Parent Settings" prompts for password
- [ ] Password entry masked (dots or asterisks)
- [ ] Incorrect password shows error message without revealing hint
- [ ] Correct password grants access to parent dashboard
- [ ] Option to change password when authenticated
- [ ] Session timeout after period of inactivity (configurable, default 10 minutes)
- [ ] Failed login attempt counter with temporary lockout after 5 failures

### Non-Functional Requirements

- [ ] Performance: Authentication check completes within 100ms
- [ ] Security: Passwords stored as salted hashes (not plain text)
- [ ] Accessibility: Password field keyboard accessible, error messages announced to screen readers
- [ ] Error handling: Graceful handling of corrupted config files

## UI/UX Requirements

### Screens Involved

- **Main Menu**: "Parent Settings" button (visible but requires auth)
- **Password Prompt Overlay**: Modal dialog covering main menu when accessed

### User Interactions

1. User clicks "Parent Settings" on main menu
2. System displays password prompt overlay
3. User types password (masked), presses Enter or clicks Submit
4. If correct: Overlay closes, parent dashboard opens
5. If incorrect: Error message displays, password field clears
6. After 5 failures: Lockout message with timer

### Visual Design Notes

- Password prompt: Dark overlay with centered dialog box
- Password field: Display dots (•) for each character
- Error message: Red text, friendly tone ("Incorrect password. Please try again.")
- Lockout message: Clear timeout display ("Too many attempts. Try again in 12:34")
- Button styles consistent with game UI (rectangular, rounded corners)

## Notes & Considerations

- **Initial Setup**: First run should have no password or a default that must be changed
- **Password Recovery**: Consider parent email for password reset (future enhancement)
- **Edge Cases**: 
  - What happens if config file is deleted? (Reset to default with warning)
  - What if game crashes during session? (Re-authentication on next access)
- **Security Note**: While not production-grade security, this prevents casual unauthorized access by children
- **Platform Consistency**: Same auth behavior on all platforms (Windows, macOS, Linux)

### Implementation Tips

- Use `getpass` module for console-based password testing
- Store config in user's home directory (`~/.word_quest/`) for portability
- Consider adding a "Show Password" checkbox for usability
- Log authentication failures (no sensitive data) for debugging

### Implementation Details

**Password Security:**
- SHA-256 hashing with random 32-byte salt
- Config stored at `~/.word_quest/auth_config.json`
- First-time setup defaults to password "parent" (prompted for change)
- Constant-time comparison not available (hashlib.compare_digest missing), using string comparison

**Session Management:**
- 10-minute inactivity timeout (configurable)
- 5 failed attempts before lockout
- 15-minute lockout duration (configurable)
- Lockout timer displays countdown to user

**UI Features:**
- Modal overlay with masked password input (• dots)
- Enter/Return to submit, Escape to cancel
- Mouse click support for buttons
- Hover effects on interactive elements
- Lockout timer display when locked
- First-time setup hint

**Key Classes:**
- `PasswordManager`: Password hashing, storage, validation
- `SessionManager`: Auth state, timeout, lockout tracking
- `AuthConfig`: Configurable timeout/lockout settings
- `PasswordPrompt`: Modal UI component for password entry
- `AuthenticatedParentDashboard`: Wrapper providing auth-gated dashboard access
